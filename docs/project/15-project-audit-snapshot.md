# 15. Аудит текущего состояния проекта (snapshot)

## Цель документа

Этот раздел фиксирует «срез» проекта в переходный период:

- целевая модель — PostgreSQL;
- DEV-контур — Supabase;
- архитектурный вектор — поэтапное разделение на Frontend + Backend (BFF).

Документ нужен как быстрая карта для подключения новых участников без повторного изучения всех 14 разделов.

---

## 1) Что уже стабильно и на что можно опираться

### 1.1 Домен и workflow

- Доменная модель и роли формализованы (admin / branch_manager / technician / controller).
- Workflow работает в двух плоскостях:
  - внешний статус `applications.status` (`IN_PROGRESS` / `COMPLETED` / `DECLINED`),
  - внутренний подстатус `applications.workflow_substatus`.
- Есть единая state-machine логика с операциями завершения/отката/согласования/отклонения.

### 1.2 Схема БД

- SQL-схема приведена в `db/reset_schema.sql` и считается «источником структуры» для PostgreSQL.
- Таблицы покрытия включают:
  - проекты/заявки/историю,
  - здания/блоки/этажи/подъезды,
  - помещения/экспликацию/МОП,
  - справочники `dict_*`,
  - версии объектов `object_versions`.

### 1.3 Документация

- Документация покрывает структуру БД, связи, проверки, роли, workflow и миграционный план.
- Для ежедневной работы достаточно трека:
  1) `01-overview-and-architecture.md`,
  2) `02-database-structure.md`,
  3) `05-workflow.md`,
  4) `13-backend-frontend-split-plan.md`,
  5) `14-iteration1-implementation.md`.

## 2) Что находится в активной миграции

### 2.1 Runtime-контур

- Frontend в DEV все еще имеет legacy-доступ к данным через Supabase SDK.
- Backend (`apps/backend`) уже реализует первую волну BFF-endpoint-ов, прежде всего для критичных workflow/lock сценариев.

### 2.2 Переход к «чистой» FE/BE-модели

- Принят поэтапный подход (Вариант A):
  1. внедрение BFF поверх текущей БД,
  2. переключение фронта на backend API по модулям,
  3. выключение direct write из frontend.

Это минимизирует риск регрессии и делает переход обратимым на промежуточных этапах.

---

## 3) Текущее разделение ответственности

### Frontend (текущий и целевой фокус)

- UI-экраны, формы, UX и локальное состояние.
- Локальные валидации удобства ввода.
- По мере миграции — отказ от прямых write-операций к БД.

### Backend BFF (усиливается в итерациях)

- Workflow-операции с серверной проверкой прав и статусов.
- Lock-механика конкурентного редактирования.
- Единая оркестрация, транзакционность и аудит.

### PostgreSQL

- Инварианты целостности: FK/UNIQUE/CHECK/индексы.
- Справочники и служебная серверная логика.

---

## 4) Практические выводы для команды

1. **Проект действительно тестовый**, и пересоздание схемы с нуля допустимо в DEV.
2. **Открытые DEV-подключения не блокируют текущую фазу**, так как приоритет — скорость итераций и перенос логики в BFF.
3. **Главный критерий готовности к следующему этапу** — уменьшение количества критичных write-path из frontend.
4. **Документация уже пригодна как спецификация**, а не только как справка: можно планировать задачи от нее напрямую.

---

## 5) Рекомендуемый рабочий workflow на ближайший период

1. Любая новая бизнес-мутация сначала проектируется как backend use-case.
2. Во frontend добавляется/расширяется API-адаптер с feature-flag переключением.
3. После стабилизации endpoint-а legacy write-path помечается deprecated.
4. Все изменения фиксируются в документации:
   - если затронута схема — `02/03`,
   - если переходы/роли — `05/09`,
   - если миграция/контракты — `13/14`.

Такой ритм обеспечивает управляемый переход к целевой архитектуре без остановки разработки.
