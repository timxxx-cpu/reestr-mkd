# 17. Transition briefing (PostgreSQL + Supabase DEV + FE/BE split)

## Что зафиксировано по текущему состоянию

- Проект находится в тестовой стадии, где **DEV-контур работает через Supabase**, а целевая платформа данных — **PostgreSQL**.
- Базовая схема и инварианты фиксируются SQL-файлом `db/reset_schema.sql`; пересоздание схемы в DEV допустимо и ожидаемо.
- Архитектура переходит к модели **Frontend + Backend (BFF)** с поэтапным отключением direct-write из frontend.

## Техническая картина перехода

1. **Данные и целостность**
   - Источник структуры: `db/reset_schema.sql`.
   - Доменные сущности уже нормализованы: проекты, заявки, здания/блоки, этажи, помещения, МОП, справочники.
   - Workflow и роли завязаны на state machine и событиях истории.

2. **Переходный runtime**
   - В DEV еще существует legacy-путь прямой записи через Supabase SDK.
   - Критичные mutation-path уже постепенно переводятся на backend endpoints под feature-flags.
   - По каждому модулю сохраняется rollback/fallback на legacy в случае регрессии.

3. **Контракты и управление рисками**
   - Для устойчивой миграции нужны стабильные DTO-контракты между DB ↔ BFF ↔ UI.
   - Основные риски: несовместимость payload, гонки при конкурентном редактировании, расхождения legacy/BFF.
   - Основные меры: транзакции, lock-aware операции, parity/smoke проверки, request tracing.

## Практическая модель работы команды

- Любые новые бизнес-мутации проектировать как backend use-case в первую очередь.
- Во frontend — только адаптер API + feature-flag переключение.
- Legacy write оставлять как временный fallback, с явным планом отключения.
- После каждой итерации синхронно обновлять документацию по схеме/ролям/workflow/контрактам.

## Что важно в тестовом контуре

- Открытые DEV-подключения и возможный reset БД здесь не считаются блокером.
- Приоритет — скорость итераций и перенос ответственности за запись/права/аудит в backend.
- Критерий зрелости этапа: сокращение direct-write путей и стабильность BFF-потока по smoke-сценариям.

## Краткий next-focus

1. Закрепить контрактный слой для registry-модулей.
2. Довести backend-first write-path до полного покрытия ключевых реестровых операций.
3. Формализовать DoD для отключения direct-write в frontend (по модульным флагам).
4. Поддерживать документацию как рабочую спецификацию миграции, а не только справку.
