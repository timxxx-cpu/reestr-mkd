# 14. Сводка по проекту и переходу на PostgreSQL/BFF

## Цель

Зафиксировать «операционное» понимание проекта на текущем этапе: что уже стабильно, что еще в переходе, и какие инварианты нельзя нарушать при дальнейшей миграции (Supabase DEV → PostgreSQL + backend-first контракты).

## 1) Что уже является устойчивой основой

1. **Схема данных и доменная декомпозиция** формализованы в `db/reset_schema.sql` и проектной документации (`docs/project/02`, `03`, `04`).
2. **Workflow-движок** уже централизован: критические переходы (complete/review/rollback/decline/restore) выполняются через backend route-слой.
3. **Frontend** использует доменные слои (`context/project/*`, `lib/api/*`) и все меньше зависит от прямых «локальных» workflow-изменений.
4. **Документационный контроль API-дрейфа** внедрен на backend через `route-manifest.json` + `docs:routes:check`.

Практический вывод: проект находится не в «хаотичной», а в **контролируемой фазе migration-by-strangler**, где legacy path планомерно вытесняется BFF-слоем.

## 2) Текущая архитектурная модель (по факту)

### Frontend (SPA)

- React + Vite, модульные редакторы по шагам.
- Основные зоны:
  - `src/features/steps/*` — пользовательские формы и матрицы;
  - `src/context/project/*` — orchestration данных/синхронизации/workflow;
  - `src/lib/*` — маппинги, валидации, вычисления, client-side utility слой.

### Backend (BFF)

- Node.js route-модули в `apps/backend/src/*`.
- Объединяет:
  - workflow-операции;
  - registry/composition CRUD;
  - read-model endpoints (`summary`, `tep-summary`, `full-registry`);
  - auth/policy/idempotency/lock semantics.

### DB

- В DEV — Supabase (PostgreSQL + RLS).
- Источник структуры — `db/reset_schema.sql`.
- Дополнительно используются lock-механизмы и аудит блокировок для предотвращения конкурентной порчи шага.

## 3) Бизнес-логика и инварианты, которые критично сохранить

### Workflow-инварианты

- Допустимые переходы строго ограничены ролью и текущим состоянием заявки.
- `application_history` обязателен как аудитный след бизнес-событий.
- Повторная подача по объекту должна учитывать активные `IN_PROGRESS` заявки.

### Данные и целостность

- Иерархия `projects → applications → steps` и `buildings → blocks → floors/entrances/units` не должна нарушаться при рефакторинге API.
- Блок-статусы шагов (`application_steps.block_statuses`) — важный контракт между editor UI и workflow gating.
- Версионирование (`object_versions`) должно сохранять ограничение на единственные `CURRENT`/`PENDING` в рамках сущности.

### Технические инварианты перехода

- Идемпотентность mutating endpoint-ов (через `x-idempotency-key`) обязательна для защиты от повторов.
- Lock acquire/refresh/release остаются обязательными для сценариев длительного редактирования.
- Любое изменение route-map должно сопровождаться синхронизацией `apps/backend/README.md` и `route-manifest.json`.

## 4) Что означает «проект постепенно переключается на backend+frontend схему»

На практике это уже выражается в следующих признаках:

1. **Source of truth workflow** — backend, а не локальная машина состояний на фронте.
2. **Read-model агрегаты** (пример: TEP summary) выносятся на backend для единообразия расчетов.
3. **Legacy/direct path cleanup** идет поэтапно, без big-bang переписывания.
4. **Контракты стабилизируются тестами** (`apps/backend/tests/*`, `tests/workflow-smoke.test.mjs`, cutover-smoke).

Это корректная стратегия для тестового проекта, где схема БД может пересобираться с нуля: важнее стабилизировать контракты и инварианты, чем сохранять временные данные DEV-среды.

## 5) Рекомендуемый порядок следующих работ

1. **Закрепить backend-first для оставшихся критичных операций** (где еще есть потенциальные обходные прямые пути).
2. **Усилить контрактные тесты на BFF** по workflow и ошибкам (особенно edge-cases rollback/reassign/decline-restore).
3. **Подготовить portable SQL-путь** (миграции/seed) отдельно от Supabase-специфики, чтобы переход на «чистый PostgreSQL» был процедурно повторяем.
4. **Вынести CI-check на docs/routes** как обязательный gate.
5. **Формализовать backend/frontend contract versioning** (минимум changelog endpoint-ов и policy по breaking changes).

## 6) Оценка риска текущего состояния

- **Низкий риск**: документация по схеме и workflow, route-manifest контроль, тестовый контур backend.
- **Средний риск**: участки, где в фронте еще остаются fallback-поведения на клиентской логике при ошибке BFF.
- **Управляемый риск**: DEV на Supabase при открытом/упрощенном подключении (для тестового стенда приемлемо, если production контур отделен).

## 7) Краткий executive summary

Проект уже находится в рабочем переходном состоянии: модель данных и workflow зрелые, BFF-контуры покрывают основную бизнес-критику, а Supabase используется как DEV-носитель PostgreSQL-схемы. Текущий приоритет — не «спасти временные данные», а завершить выравнивание контрактов frontend↔backend и закрепить это тестами/CI-процессами.
