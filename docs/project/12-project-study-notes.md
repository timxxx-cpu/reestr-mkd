# 12. Конспект изучения проекта (стартовый baseline)

## Цель документа

Зафиксировать текущее понимание проекта перед постановкой следующих задач:

- архитектура;
- доменная модель и БД;
- workflow и роли;
- ключевая логика валидаций;
- миграционный контекст (Supabase DEV → PostgreSQL target).

## 1) Что это за система

`Реестр МКД` — это система поэтапного ведения данных о проекте/заявке и объекте (МКД) с привязкой к бизнес-процессу согласования:

- создается `project` + связанная `application`;
- данные наполняются по шагам (`application_steps`) через редакторы;
- заявки проходят через роли (техник, контролер, менеджер, админ);
- результат: завершение заявки (`COMPLETED`) либо отказ (`DECLINED`) с аудиторным следом.

## 2) Архитектурная картина

### Frontend

- SPA на React/Vite.
- Выделены слои API (`src/lib/api/*`, `src/lib/api-service.js`), workflow-утилиты и state-machine, контексты `ProjectContext` + domain hooks.
- Большая часть предметной логики разнесена по:
  - `src/context/project/*` (data/sync/workflow layers),
  - `src/lib/*` (маппинг, валидации, вычисления, идентификаторы),
  - `src/features/steps/*` (шаговые редакторы).

### Backend/DB (на текущем этапе)

- DEV-контур работает через Supabase (PostgreSQL + RLS).
- Базовая схема и reset-подготовка фиксируются SQL-скриптом `db/reset_schema.sql`.
- Есть RPC-функции для конкурентного редактирования (lock acquire/refresh/release).

## 3) Базовые сущности и ключевые таблицы

### Core

- `projects` — карточка проекта/объекта (адрес, кадастр, интеграционные данные, UJ-код).
- `applications` — жизненный цикл заявки (статус, substatus, текущий этап/шаг, исполнитель, история отказов).
- `application_history` — аудит переходов статусов/операций.
- `project_participants`, `project_documents` — участники и документы.

### Workflow & concurrency

- `application_steps` — состояние шагов и блоков (включая JSONB статусы блоков).
- `application_locks`, `application_lock_audit` — защита от конкурентного редактирования + аудит.

### Building domain

- Иерархия: `buildings` → `building_blocks` → `floors` / `entrances` / `units`.
- Доп. ветки: конструктив/инженерия, подвалы, МОП, паркинг (`basement_parking_levels`), помещения и др.

### Dictionaries

- `dict_*` наборы (типы помещений, конструктив, статусы и т.п.) + seed-подход для DEV.

### Versioning

- `object_versions` + `dict_version_statuses`.
- Инварианты: не более одной `PENDING` и одной `CURRENT` версии на объект.

## 4) Workflow (операционный смысл)

Система опирается на этапно-шаговую модель:

- глобальный статус заявки: как минимум `IN_PROGRESS` → (`COMPLETED` | `DECLINED`);
- детализация через `workflow_substatus` и `current_step/current_stage`;
- ключевые операции процесса:
  - `COMPLETE_STEP`;
  - `ROLLBACK_STEP`;
  - `REVIEW_APPROVE`;
  - `REVIEW_REJECT`;
  - переназначение технику;
  - повторная подача с проверкой активной заявки по объекту.

Практический эффект:

- прозрачный аудит действий;
- контролируемые переходы между состояниями;
- отделение исполнительского ввода и контрольного решения.

## 5) Валидации и контроль качества данных

Валидации многоуровневые:

- базовые логические проверки (балансы, непротиворечивость, обязательные поля);
- шаговые валидаторы по разделам (композиция, реестры, этажность, подъезды, помещения, МОП, паркинг);
- фиксация статусов заполнения блоков на шаге сохранения.

В UI предусмотрены типы сообщений (`error`, `warning`, `info`) и вывод по контексту шага/блока.

## 6) Интеграция, синхронизация, миграционный контекст

Текущее состояние проекта соответствует переходному периоду:

- DEV-эксплуатация через Supabase;
- целевое направление — полноценный PostgreSQL-контур;
- существует документированный UI↔DB mapping и миграционные заметки.

Критичные моменты для следующих задач:

1. Не ломать контракт маппинга DTO/UI/DB.
2. Не нарушать workflow-инварианты при изменениях шагов.
3. Учитывать lock-механику при операциях записи.
4. Проверять совместимость с versioning (`PENDING`/`CURRENT` уникальности).

## 7) Что уже можно считать стабильными опорами

- Схема БД описана достаточно полно и пригодна к пересборке с нуля.
- Ролевая модель и шаговый процесс формализованы.
- Есть отдельный слой для интеграции и migration notes.
- Есть явные правила версионирования доменных объектов.

## 8) Базовый подход к будущим задачам

Предлагаемый рабочий шаблон для любых следующих изменений:

1. Определять, это изменение:
   - только UI,
   - UI+mapping,
   - workflow,
   - schema/API.
2. Для изменений данных — сразу фиксировать:
   - затронутые таблицы/поля,
   - влияние на валидаторы,
   - влияние на шаговые статусы и историю.
3. Для изменений workflow — проверять:
   - права роли,
   - допустимость перехода,
   - аудит и откат.
4. Для Postgres-миграции — держать совместимость по seed/справочникам и уникальным ограничениям.
