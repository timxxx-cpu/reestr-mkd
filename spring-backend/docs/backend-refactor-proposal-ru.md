# Предложения по глобальному рефакторингу Spring backend

## Что видно по текущей базе кода

- В проекте есть несколько очень больших контроллеров: `ProjectExtendedController` (~1463 строки), `RegistryController` (~1271 строка), `ProjectController` (~774 строки).
- Контроллеры совмещают HTTP-слой, бизнес-логику, SQL, маппинг и валидацию в одном классе.
- Широко используется `JdbcTemplate` прямо в API-контроллерах, что усложняет транзакционность, тестирование и повторное использование логики.
- В ряде мест используется динамический `Map<String, Object>` вместо строгих request/response DTO.
- Доля тестов пока низкая относительно объёма production-кода (примерно 6 тестовых классов на 154 production-класса).

## Цели рефакторинга (target state)

1. **Тонкие контроллеры**: только валидация запроса, вызов use-case сервиса, формирование ответа.
2. **Сервисный слой по use-case**: один сервис = один сценарий (например `SyncParkingPlacesUseCase`, `SaveBuildingDetailsUseCase`).
3. **Разделение доменов** (`project`, `registry`, `workflow`, `integration`) без «протекания» логики между пакетами через контроллеры.
4. **Предсказуемые контракты API**: typed DTO + Bean Validation + единый формат ошибок.
5. **Повышение тестопригодности**: unit тесты use-case сервисов + slice/integration тесты ключевых эндпоинтов.

## Приоритетные проблемы и предложения

## 1) Декомпозиция перегруженных контроллеров

**Проблема**  
`ProjectExtendedController` и `RegistryController` содержат большую часть бизнес-логики и SQL, из-за чего изменения рискованные и дорогие.

**Предложение**
- Разбить на feature-controller’ы по bounded context:
  - `ProjectContextController`
  - `ProjectBuildingDetailsController`
  - `RegistryParkingController`
  - `RegistryBlocksController`
  - `RegistryUnitsController`
- Вытащить бизнес-правила в use-case классы в `domain/*/service/usecase`.
- Ограничить контроллеры 100–250 строками как инженерный стандарт.

**Выигрыш**
- Понятнее ownership.
- Проще code review.
- Ниже вероятность side effects.

## 2) Убрать SQL из REST-контроллеров

**Проблема**  
Прямые SQL-запросы в контроллерах увеличивают связность и дублирование.

**Предложение**
- Вынести SQL в repository/dao слой:
  - `registry/repository/queries/*`
  - `project/repository/queries/*`
- Для сложных выборок оставить `JdbcTemplate`, но обернуть в специализированные query-сервисы с явными методами.
- Для write-операций по возможности перейти на агрегаты/репозитории, сохранив JDBC только для batch/сложных SQL.

**Выигрыш**
- Прозрачные транзакционные границы.
- Удобнее покрывать тестами.
- Проще мигрировать/оптимизировать SQL.

## 3) Типизировать вход/выход API

**Проблема**  
Много `MapPayloadDto` и `Map<String,Object>` затрудняют валидацию, автодокументацию и обратную совместимость.

**Предложение**
- Для всех mutable эндпоинтов сделать request DTO с `jakarta.validation`.
- Для ответов использовать явные response DTO вместо map-пейлоадов.
- Ввести отдельный mapper слой (например MapStruct) для перехода entity -> dto.

**Выигрыш**
- Раннее обнаружение ошибок входных данных.
- Стабильные API-контракты.
- Лучшая поддержка OpenAPI/Swagger.

## 4) Унифицировать обработку ошибок

**Проблема**  
Есть `ApiException` и глобальный handler, но в доменной логике пока смешаны инфраструктурные и бизнес-ошибки.

**Предложение**
- Ввести иерархию доменных исключений (`DomainException`, `NotFound`, `Validation`, `Forbidden`, `Conflict`).
- Нормализовать error code каталог и связывать его с HTTP-статусами в одном месте.
- Добавить correlationId/requestId в error response для трассировки.

## 5) Перестроить security слой

**Проблема**  
`AuthContextFilter` делает ручной парсинг JWT внутри фильтра, логика верификации и маппинга claims tightly coupled.

**Предложение**
- Перейти на `oauth2-resource-server` + `JwtDecoder` (или оставить custom decoder, но вынести в отдельный `TokenService`).
- Разделить responsibilities:
  - токен-верификация,
  - резолв principal,
  - авторизация policy.
- Политики доступа хранить как явно типизированные правила (enum/scopes), а не строковые magic values.

**Выигрыш**
- Меньше security-рисков.
- Проще ревью и аудит.
- Легче внедрять дополнительные auth-провайдеры.

## 6) Тестовая стратегия на 3 уровня

1. **Unit**: use-case сервисы (быстро, изолированно).
2. **Slice (`@WebMvcTest`, `@DataJpaTest`)**: контракты API и репозиториев.
3. **Integration**: критические happy/sad path (лучше с Testcontainers PostgreSQL).

### KPI качества после рефакторинга
- Контроллеры >500 строк: **0**.
- Покрытие unit + integration: **минимум 60%** в доменных сервисах.
- Все POST/PUT/PATCH имеют DTO + Bean Validation: **100%**.
- Количество business SQL в контроллерах: **0**.

## Предлагаемый roadmap (по этапам)

### Этап 0 (1 неделя): подготовка
- Архитектурные правила (ADR + package conventions).
- Feature flags для безопасного поэтапного переноса.
- Базовые метрики (размер классов, время ответа, ошибки 5xx/4xx).

### Этап 1 (1–2 недели): быстрая декомпозиция API-слоя
- Разделить крупнейшие контроллеры на 4–6 целевых контроллеров.
- Без изменения бизнес-логики (mechanical refactor).

### Этап 2 (2–4 недели): вынос use-case логики и SQL
- По одному сценарию за раз переносить в сервисы и query-компоненты.
- Добавлять unit тесты на каждый вынесенный use-case.

### Этап 3 (1–2 недели): DTO/валидация + error model
- Typed DTO и валидация на все write-endpoints.
- Единый каталог ошибок.

### Этап 4 (1–2 недели): security hardening
- Декомпозиция токен-обработки.
- Интеграционные security тесты.

### Этап 5 (постоянно): производительность и observability
- Логирование по requestId.
- Метрики slow SQL/slow endpoint.
- Профилирование hot paths.

## Быстрые практичные win’ы (можно начать сразу)

1. Вынести из `RegistryController` операции parking sync в `ParkingService` + unit тесты.
2. Вынести из `ProjectExtendedController` блок сохранения building details в `BuildingDetailsService`.
3. Заменить `MapPayloadDto` в наиболее критичных POST/PUT на строгие DTO.
4. Ввести архитектурный линтер-правило: запрет `JdbcTemplate` в `..api..` пакетах.

## Риски и как их снижать

- **Риск регрессий**: переносить по use-case, не «большим взрывом».
- **Риск поломки фронта**: сначала contract tests + consumer-driven checks для текущих endpoint contract.
- **Риск перетягивания сроков**: фиксировать Definition of Done для каждого этапа (код + тесты + метрики).

## Рекомендуемый целевой техстек практик

- Java 21 + Spring Boot 3 (сохраняем).
- OpenAPI генерация контрактов.
- MapStruct для маппинга DTO.
- Testcontainers PostgreSQL для интеграционных тестов.
- ArchUnit для архитектурных ограничений (например, запрет SQL в контроллерах).
